The Waterfall.
Seven stages. Fail-closed.

Every action proposed to Project-AI passes through seven sequential stages before any effect is permitted. A failure at any stage is a permanent denial for that request. There is no retry path, no escalation route, no stage that can be skipped. The pipeline is closed.

The Seven Stages

Each stage, in full.

Stages execute in strict order. A HALT at any stage terminates the pipeline — no subsequent stages run. Every verdict is sealed to the CCS before moving forward.

0
Stage 0 · Structural Validation

Schema, format, and well-formedness

The request must conform to the PSIA request schema. Missing fields, unknown types, invalid encodings, and malformed envelopes are rejected immediately. No processing begins until the request is structurally sound.

src/psia/waterfall/stage_0_structural.py
PASS → continue
FAIL → HALT
1
Stage 1 · Signature Verification

Cryptographic identity of the requestor

Every request must be signed with an ed25519 key registered in the PSIA identity ledger. Unsigned requests, invalid signatures, expired keys, and revoked identities all produce a HALT. There is no anonymous request path.

src/psia/waterfall/stage_1_signature.py
PASS → continue
FAIL → HALT
2
Stage 2 · Behavioral Analysis

Pattern matching and anomaly detection

The request is evaluated against behavioural baselines: access frequency, resource scope, deviation from historical patterns. Anomalies above threshold are flagged and escalated to the PSIA quorum for review before proceeding.

src/psia/waterfall/stage_2_behavioral.py
PASS → continue
ANOMALY → quorum
3
Stage 3 · Shadow Execution

Dry-run in an isolated execution sandbox

The proposed action is executed speculatively in a Shadow Thirst environment with no external effects. The shadow output is compared against the model's predicted output. Divergence above tolerance triggers a HALT — this catches subtle goal misalignment before any real effect occurs.

src/psia/waterfall/stage_3_shadow.py · src/psia/shadow/operational_semantics.py
CONVERGE → continue
DIVERGE → HALT
4
Stage 4 · Gate Evaluation (OCEE)

AGI Charter invariants + Four Laws + OCEE constraint lattice

The full OCEE constraint lattice evaluates the request: all five invariant gates (safety.human_life, safety.no_harm, safety.civilisational, audit.seal_required, authority.human_instruction) plus the Four Laws constitutional gate. All must hold. A HALT at any gate propagates immediately.

src/psia/waterfall/stage_4_gate.py · src/cerberus/ocee/engine.py
ALL HOLD → continue
ANY HALT → HALT
5
Stage 5 · Commit Authorisation

Quorum engine and capability grant

The PSIA quorum engine reviews the request against the active capability set. Actions requiring elevated capability must have a valid, non-expired capability grant. The Triumvirate's unanimous approval is required for capability grants not already on the active list. A pending quorum vote holds the request here.

src/psia/waterfall/stage_5_commit.py · src/psia/gate/quorum_engine.py
AUTHORISED → continue
NO GRANT → HALT
6
Stage 6 · Memory Recording (CCS)

Append-only seal — the last act before execution

The full decision record — request, all stage verdicts, quorum state, capability grant, timestamps — is sealed to the Constitutional Code Store with an ed25519 signature and RFC 3161 timestamp. If the seal fails for any reason, the action does not execute. The audit trail is a precondition, not an afterthought.

src/psia/waterfall/stage_6_memory.py · src/psia/canonical/ledger.py
SEALED → execute
SEAL FAIL → HALT
Design Principle

Why fail-closed?

Failure is not ambiguous.

A system that fails open permits actions that should be denied whenever it encounters an unexpected state. In a sovereign AI context, unexpected state is precisely when the constraint is most needed. Fail-closed means the constraint holds even when the system cannot reason about its situation.

Seven stages because each catches a different class of threat.

Structural corruption (Stage 0) is distinct from identity forgery (Stage 1), which is distinct from goal misalignment (Stage 3), which is distinct from charter violation (Stage 4). Merging stages would create blind spots. Separation is a safety property.
