THREE DEPLOYMENT METHODS:

  Method 1 (Simple):
    helm install project-ai ./helm/project-ai \
      -f helm/values.prod.yaml \
      --set-string secrets.api.token="token-here"

    Pros: Simple, no prerequisites
    Cons: Token in CLI history, requires token at deploy time
    Use: Development, testing

  Method 2 (Recommended):
    kubectl create secret generic project-ai-api-secrets \
      --from-literal=PROJECT_AI_API_TOKEN="token-here"

    helm install project-ai ./helm/project-ai \
      -f helm/values.prod.yaml \
      --set secrets.create=false

    Pros: Token not exposed in CLI, pre-created separately
    Cons: Two-step process
    Use: Production with manual secret management

  Method 3 (Enterprise):
    # Deploy External Secrets Operator
    # Create SecretStore (Vault credentials)
    # Create ExternalSecret (references Vault)

    helm install project-ai ./helm/project-ai \
      -f helm/values.prod.yaml \
      --set secrets.create=false

    Pros: Secrets never in cluster, automatic rotation
    Cons: Requires external system (Vault, AWS Secrets Manager)
    Use: Production with centralized secret management
