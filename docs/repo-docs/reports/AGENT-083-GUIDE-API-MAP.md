## 📋 Link Types & Patterns

### 1. Class/Function Reference Links

**Pattern**: `[[module-name#ClassName|ClassName]]`

**Example**:
```markdown
The system uses [[ai_systems#AIPersona|AIPersona]] class to manage personality traits.
```

### 2. API Reference Sections

**Pattern**: Add "## API Reference" section at end of tutorial

**Example**:
```markdown
## API Reference

### Core Classes

- **[[ai_systems#AIPersona|AIPersona]]** - Main personality management class
  - `update_trait(trait, value)` - Modify personality trait
  - `get_mood()` - Get current mood state
  - `_save_state()` - Persist personality to disk

- **[[ai_systems#FourLaws|FourLaws]]** - Ethics validation framework
  - `validate_action(action, context)` - Check action against Four Laws
  - `get_hierarchy()` - Get law hierarchy order

### Related Modules

- [[user_manager|UserManager]] - User authentication and profiles
- [[command_override|CommandOverride]] - Master password system
- [[intelligence_engine|IntelligenceEngine]] - OpenAI integration
```

### 3. Inline Documentation Links

**Pattern**: Link when mentioning API concepts

**Example**:
```markdown
Before proceeding, the system validates the action using [[ai_systems#FourLaws.validate_action|FourLaws.validate_action()]].
```

### 4. Method/Attribute Links

**Pattern**: `[[module#Class.method|method()]]`

**Example**:
```markdown
Call [[user_manager#UserManager.authenticate_user|authenticate_user()]] with credentials.
```

## 📐 Quality Standards

### Link Quality Gates

✅ **MUST HAVE**:
1. Every tutorial mentions at least 3 API modules → each gets wiki link
2. Every code snippet showing class usage → class name linked
3. Every feature description → related module linked
4. API Reference section at end of every tutorial

❌ **AVOID**:
1. Linking same class/method multiple times in same paragraph
2. Links to non-existent API docs (check first!)
3. Generic links without specific class/method anchors
4. Circular references (A→B→A)

### Link Validation Checklist

- [ ] Link target exists (module/class is real)
- [ ] Anchor syntax correct (`#ClassName.method_name`)
- [ ] Context makes sense (not just spamming links)
- [ ] First mention of API element is linked
- [ ] API Reference section comprehensive
- [ ] No dangling references (incomplete links)

## 📝 Notes & Decisions

### Design Decisions

1. **Wiki Link Syntax**: Use Obsidian-style `[[...]]` for consistency
2. **Anchor Tags**: Use `#ClassName.method_name` format
3. **Link Placement**: First mention only (avoid clutter)
4. **API Sections**: Always at end of tutorial
5. **Module Granularity**: Link to specific classes/methods, not just files

### Known Challenges

1. **Multi-class Files**: `ai_systems.py` contains 6 classes - need precise anchors
2. **External Dependencies**: actionlint API docs are external (GitHub)
3. **Generated Docs**: Some API docs may not exist yet
4. **Link Maintenance**: Need process to update links when APIs change

### Future Enhancements

1. **Automated Link Checking**: CI/CD pipeline to validate links on PR
2. **Link Usage Analytics**: Track which API links are most clicked
3. **Smart Link Suggestions**: AI-powered link recommendations in tutorials
4. **Bidirectional Link Graph**: Visualize tutorial↔API relationships
